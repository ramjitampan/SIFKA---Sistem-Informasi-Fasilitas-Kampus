# Frontend Development Guidance: SIFKA API Integration

This document provides a technical roadmap for frontend developers integrating with the SIFKA API.

## 🔐 Authentication (Laravel Sanctum)

<details markdown="1">
<summary>Click to see details</summary>

The API uses Bearer Token authentication via Laravel Sanctum.

- **Login/Register:** Standard POST requests. On success, the API returns an `access_token`.
- **Authorization:** Include the header `Authorization: Bearer <token>` in all protected requests.
- **Throttling:** Auth endpoints are limited to **5 requests per minute**. Implement exponential backoff or user-facing alerts for `429 Too Many Requests`.

</details>

## 📍 Data Conventions: Coordinates

<details markdown="1">
<summary>Click to see details</summary>

A key architectural feature of SIFKA is the **Asymmetric Coordinate Mapping**.

### Sending Data (Input)
When creating or updating Buildings, Facilities, or Reports, you can send flat parameters:
```json
{
  "name": "Library",
  "latitude": -0.9482,
  "longitude": 100.3606
}
```

### Receiving Data (Output)
The API always returns coordinates in a structured `coordinate` object. This is optimized for mapping libraries (Leaflet, Google Maps, etc.):
```json
{
  "data": {
    "name": "Library",
    "coordinate": {
      "lat": -0.9482,
      "lng": 100.3606
    }
  }
}
```

</details>

## 🔍 Global Search Integration

<details markdown="1">
<summary>Click to see details</summary>

SIFKA uses **Meilisearch** for ultra-fast, fuzzy searching.

### Endpoints
- `GET /api/buildings/search?q={query}`
- `GET /api/categories/search?q={query}`
- `GET /api/facilities/search?q={query}`
- `GET /api/reports/search?q={query}` (Filtered by Role)
- `GET /api/users/search?q={query}` (Admin Only)

### Search Features
- **Fuzzy Matching:** Typo-tolerant search.
- **Rich Context:** Searching for a facility name will also match its Building or Category name.
- **Speed:** These endpoints are optimized for real-time "search-as-you-type" UI components.

</details>

## 📡 Real-time Updates (Laravel Reverb)

<details markdown="1">
<summary>Click to see details</summary>

SIFKA uses **Laravel Reverb** for low-latency, WebSocket-driven updates.

### Key Events
- **`ReportUpdated`**: Broadcasted on the `private-reports` channel whenever a report status or location changes.

### Integration Requirements
- **Dependencies**: `laravel-echo`, `pusher-js`.
- **Channel Authorization**: Since the reports channel is private, the client must be authenticated. Echo will automatically use the `Authorization: Bearer <token>` header if configured correctly.

### Frontend Configuration (Example)
```javascript
import Echo from 'laravel-echo';
import Pusher from 'pusher-js';

window.Pusher = Pusher;
window.Echo = new Echo({
    broadcaster: 'reverb',
    key: import.meta.env.VITE_REVERB_APP_KEY,
    wsHost: import.meta.env.VITE_REVERB_HOST,
    wsPort: import.meta.env.VITE_REVERB_PORT ?? 80,
    wssPort: import.meta.env.VITE_REVERB_PORT ?? 443,
    forceTLS: (import.meta.env.VITE_REVERB_SCHEME ?? 'https') === 'https',
    enabledTransports: ['ws', 'wss'],
});
```

</details>

## ⚠️ Error Handling

<details markdown="1">
<summary>Click to see details</summary>

- **422 Unprocessable Entity:** Validation failure. The response will contain an `errors` object mapping fields to error messages.
- **403 Forbidden:** Authorization failure. Occurs if a Student tries to access Admin features.
- **401 Unauthorized:** Token expired or missing.

</details>
